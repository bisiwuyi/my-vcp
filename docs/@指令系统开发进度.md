## 一、需求概述

### 1.1 目标
将 VCPChat 聊天窗口升级为"自然语言高级命令终端"，通过 `@plugin/command 参数` 格式实现确定性的插件调度。

### 1.2 两阶段搜索设计
- **阶段1：** 输入 `@Op` → 搜索插件名 → 显示插件列表
- **阶段2：** 输入 `@OpenCli/` → 搜索该插件下的命令 → 显示命令列表

### BUG-12：fuzzyMatch 评分算法缺陷导致所有插件搜索失败 🔴
- **文件：** `inputEnhancer.js:471-488`
- **问题：** `fuzzyMatch` 有两个严重缺陷：(1) `target.includes(query)` 在 `target.startsWith(query)` **之前**执行，由于 includes 包含 startsWith，startsWith 分支永远不会触发，且 includes 返回的分数 `query.length / target.length` 对短查询极低（如 `O` 对 `OpenCli` = 0.143）；(2) 所有单字母查询的模糊匹配分数都低于 0.3 阈值被拒绝。导致用户输入 `@O` 后插件搜索返回空结果，fallback 到 @note 搜索 Notemodules 目录，弹出的是文档而非插件列表
- **修复：** 重写评分逻辑——精确匹配=1.0, 前缀匹配=0.9, 包含匹配=0.3+0.6×(查询长/目标长), 模糊子序列保持不变
- **版本：** v1.2.1

### BUG-14：pluginRegistryManager.init() 时序错误导致搜索永远返回空 🔴
- **文件：** `pluginRegistryManager.js:156-160`
- **问题：** `pluginRegistryManager` 作为普通 `<script>` 在 `renderer.js`（`type="module defer"`）之前加载。其 `init()` 在 `renderer.js` 的 `DOMContentLoaded` 回调执行时就被调用，但此时 `window.globalSettings` 还是空对象 `{}`（要等 `loadAndApplyGlobalSettings()` 完成后才被填充）。导致 `fetchPluginRegistry()` 读取 `vcpServerUrl` 为 `undefined`，直接返回 `null`，所有搜索失败。弹出层因搜索结果为空而隐藏。
- **修复：** 移除 `init()` 中的 `fetchPluginRegistry()` 预抓取调用，改为首次搜索时再抓取。添加详细日志方便排查。
- **版本：** v1.3.0

### BUG-15：全局作用域 `let electronAPI` 重复声明导致 SyntaxError 🔴

### BUG-21：forceRoute 时走错流式分支导致 stream 错误 🔴🔴
- **文件：** `chatHandlers.js:765`，`vcpClient.js:285-385`
- **问题：** 前端发送 `stream: true` 的请求，后端 `vcp_force_route` 永远返回 JSON（`Content-Type: application/json`），但 `vcpClient.js` 根据 `modelConfig.stream === true` 走流式分支，`response.body` 被当作 SSE 读取，JSON 响应解析失败导致 `processStream()` 内部抛出异常，返回 `{ streamError: true }`，前端显示"请求流式回复失败"。
- **修复A：** `chatHandlers.js` 在检测到 `forceRouteInfo` 时强制 `stream: false` 注入到 requestBody。
- **修复B：** `vcpClient.js` 在第 285 行前加 forceRoute 模式检测（正则 `/^\s*@[\w\u4e00-\u9fa5\-]+\/[\w\u4e00-\u9fa5\-]+/`），检测到时强制走非流式分支。
- **版本：** v1.4.0

### BUG-22：selectCmdSuggestion 文本替换逻辑错误导致内容污染 🔴

- **文件：** `chatManager.js:1022-1118`
- **问题：** `response.type === 'vcp_force_route_result'` 的处理逻辑嵌套在 `if (!useStreaming)` 块内，而 `useStreaming` 默认为 `true`（除非 Agent 配置显式关闭）。即使 vcpClient 已修复为在 forceRoute 时走非流式返回 `{ response: {...}, context }`，chatManager 在流式模式下根本不会解构 `vcpResponse.response`，导致 forceRoute 结果无法被识别和处理。

**BUG-26：** inputEnhancer input handler 查询末尾空格导致搜索失败 🔴
- **文件：** `inputEnhancer.js:373`
- **问题：** `selectCmdSuggestion` 在 plugin-only 选择时插入 `@PluginName/`（无尾部空格），但 command 选择时插入 `@PluginName/Command `（尾部有空格）。当用户刚选完命令、input 事件触发搜索时，若 textarea 内容为 `@OpenCli/generate `（尾部空格），`atMatch[1]` 捕获 `"OpenCli/generate "`（含尾部空格），传递给 `searchPluginCommands` 的 query 含尾部空格，导致 fuzzyMatch 在 plugin 匹配阶段失败（`"opencli/generate "` vs `"opencli"` 匹配度为 0）。

- **版本：** v1.4.2

**BUG-27：** 选择插件后按回车消息被立即发送，AI 不知道用户要做什么 🔴🔴
- **文件：** `inputEnhancer.js:413-417`
- **问题：** `inputEnhancer` 的 keydown handler 中 Enter 分支调用 `e.preventDefault()` 但没有 `e.stopImmediatePropagation()`，导致 `chatManager` 注册的 keydown handler（`event-listeners.js:395-399`）仍然会执行并调用 `handleSendMessage()`。由于 `selectCmdSuggestion` 是 `async` 函数（虽然无 await），click handler 返回后 keydown handler 立即返回，然后 `chatManager.handleSendMessage` 捕获的是旧的 `content`（"@Op"），消息被错误发送。
- **修复A：** 去掉 `selectCmdSuggestion` 的 `async` 关键字，确保同步执行完毕后 keydown handler 才返回。
- **修复B：** 在 Enter 分支添加 `e.stopImmediatePropagation()`，阻止 `chatManager` 的 keydown handler 触发。
- **版本：** v1.4.3

## 三、功能优化（v1.5.0）

### 功能-1：System Prompt 动态 Schema 注入
- **文件：** `chatCompletionHandler.js:627-803`
- **改动：** 在构造 LLM 参数提取的 system prompt 时，调用 `getPluginRegistry()` 读取目标插件和命令的 `description`（参数说明）和 `example`（格式示例），注入 prompt 中为 LLM 提供精确的参数格式引导。
- **效果：** LLM 能更准确地将用户自然语言映射到目标工具的参数结构（如 TavilySearch 的 query/topic/max_results 等）。
- **版本：** v1.5.0

### 功能-2：多阶段解析 + 静默 Fallback
- **文件：** `chatCompletionHandler.js:627-803`
- **改动：** 实现三阶段解析策略：(1) 严格正则（要求完整 `<<<[TOOL_REQUEST]>>>...<<<[END_TOOL_REQUEST]>>>` 闭合块）(2) 宽松正则（只要求开始标记，捕获到 `>>>` 或行尾）(3) 静默 Fallback（解析失败时用 `user_input` 直接作为 `prompt` 参数执行工具）。同时 LLM 调用失败时也触发 Fallback 而非直接报错。
- **效果：** 大幅降低"参数解析失败"错误，工具总能执行（即使 LLM 输出格式不完全正确），用户体验更流畅。
- **版本：** v1.5.0

## 三、v1.6.0 重大架构重构

### 核心变化：工具直连 → Agent 解读

**旧流程（v1.5.x）：** 检测到 `@指令` → 专用 LLM (forceRouteModel) 提取参数 → 直接执行工具 → 返回原始结果

**新流程（v1.6.0）：** 检测到 `@指令` → 从 Registry 获取工具 schema → 注入自然语言提示 → Agent LLM 自行决定如何调用 → VCP Loop 执行 → Agent 解读结果 → 返回

### 功能-3：移除 Fallback，完全依赖 Agent LLM
- **文件：** `chatCompletionHandler.js`
- **改动：** 删除 forceRouteModel 调用、Fallback 逻辑、`parseToolRequestBlock()`、`extractParams()`、`getCommandSchema()` 等全部旧逻辑

### 功能-4：区分"只选插件"vs"选插件+命令"两种注入策略
- **文件：** `chatCompletionHandler.js`
- **改动：** `buildForceRouteHint()` 函数
  - `command` 有值 → 注入**该命令的** description + example
  - `command` 为空 → 注入**插件级别描述 + 所有命令列表**，让 Agent 帮用户选择命令
- **版本：** v1.6.0

### 功能-5：前端移除 forceRouteModel 配置
- **文件：** `main.html`、`global-settings-manager.js`、`renderer.js`、`vcpClient.js`、`chatHandlers.js`
- **改动：** 
  - 删除 `@指令 参数解析模型` 设置 UI
  - 删除 `forceRouteModel` 保存/加载逻辑
  - 删除 `vcp_force_route` 中的 `model` 字段
  - 删除 `vcpClient.js` 中的 `detectForceRouteCommand()` 调用（已无用）
  - 移除 `stream: false` 覆盖逻辑（不再需要）
- **版本：** v1.6.0

```json

  "messages": [...],
  "vcp_force_route": {

  "type": "force_route_status",
  "status": "executing|success|failed",
  "plugin": "OpenCli",

}
```
